# 1. Role of pricing-metrics-aggregator in the platform

## Status

Accepted — pricing-metrics-aggregator is the Pricing Metrics Aggregator container from the C4 diagram. It polls the funnel-events MinIO bucket on a rolling window (5 minutes default), computes aggregates over `search.v1` and `booking.v1` events, and publishes them to Prometheus as last-window gauges. Feeds Grafana dashboards and AlertManager rules downstream; is fed by markup-svc + funnel-sim upstream.

## Context

pricing-observability's Analytics/BI arc (ADR-0021) shipped Jupyter notebooks that compute conversion, walkoffs, GMV, and elasticity on demand. Two use cases the notebooks do NOT serve:

1. **Live Grafana dashboards.** A Price Manager asking "what's conversion right now?" or "did GMV drop in the last hour?" needs a metric that Grafana can plot over time. Notebooks answer point-in-time questions.
2. **On-call alerting.** AlertManager needs Prometheus rules like `rate(pricing_purchases_last_window[15m])` to fire when conversion collapses. Notebooks cannot drive alerts.

Both are the domain of the Pricing Metrics Aggregator container. This service is the bridge from the batched Decision Event Stream in MinIO to the continuous Prometheus surface Grafana + AlertManager consume.

### Alternatives considered
- **Extend markup-svc / funnel-sim with their own aggregators.** Rejected: each service has its own domain — pricing decisions vs. funnel events — and the aggregation is fundamentally a JOIN across both. Colocating breaks the port/adapter boundary those services carefully maintain.
- **Use Prometheus's `recording_rules` to derive metrics from existing counters.** Rejected: markup-svc's `markup_decide_total` counter and funnel-sim's `funnel_event_sink_flushed_total` counter are per-emission, per-instance, per-key-prefix; joining them at Prometheus's query layer doesn't produce a conversion rate because there's no shared cardinality between "one decision" and "one purchase" in a way Prometheus can reason about.
- **Push the aggregation to the DuckDB notebooks + schedule notebook execution via cron.** Rejected: latency to Grafana is >5 min (cron granularity + notebook boot); no clean path to alerting; every rescheduled run reloads the whole compose stack's worth of data. A purpose-built service that keeps the S3 client warm and only reads incremental partitions is orders of magnitude cheaper.
- **Cloud-hosted managed aggregator (Amazon Athena scheduled queries, GCP Dataflow, etc.).** Rejected for the platform-locally-runnable requirement: `docker compose up` should bring up the whole stack including aggregation.

### Choice
A small Go service, mirrors the funnel-sim scaffold pattern. minio-go SDK reads the buckets on a tick; per-window aggregates accumulate in-memory; Prometheus /metrics exposes gauges plus one `runs_total` counter for meta-observability.

## Decision

### Boundaries

- **In-scope for v0.0.1**: aggregate over search.v1 impressions and booking.v1 (purchased / timeout) with `{env}` cardinality only. Six last-window gauges + run-outcome counter + run-duration histogram.
- **Out-of-scope for v0.0.1 (parked with concrete triggers)**: per-arm (`experiment`, `variant`), per-segment (`customer_tier`, `country`), per-rule aggregates. Requires ADR-0003 (cardinality caps) — see Not-closed.
- **Out-of-scope permanently**: real-time streaming. This service polls; consumers who need sub-minute latency should look at OpenTelemetry span metrics or shape a Kafka/Pulsar substrate (which is a Learning Loop follow-on, not this service's concern).

### The published metric surface

```
# Last-window gauges — overwritten every tick
pricing_impressions_last_window{env}
pricing_purchases_last_window{env}
pricing_walkoffs_at_init_last_window{env}
pricing_walkoffs_at_reserve_last_window{env}
pricing_gmv_last_window_eur{env}
pricing_conversion_rate_last_window{env}

# Meta-observability
pricing_metrics_aggregator_runs_total{env, outcome="ok"|"error"}    # counter
pricing_metrics_aggregator_run_duration_seconds{env}                 # histogram
```

Gauges are "last-window snapshots" — a Grafana panel plots them over time; an AlertManager rule looks at their delta. Counters and histograms carry the aggregator's own health signals.

### Aggregation loop

- On boot: bind /metrics, /healthz, /readyz immediately so compose health checks settle. Serve zeros as the first metric read.
- After `--warmup` (5s default): run the first pass so /metrics reflects the actual bucket state.
- Every `--interval` (5m default): pass over the last `--interval` of events.
- The window is **tumbling** — `[now - interval, now]` per pass. Contiguous windows do not overlap; every event is counted at most once across the stream.
- Loop is single-goroutine (goroutine driving the ticker; HTTP served by the mux goroutine pool). No shared state between them beyond the Prometheus registry, which is thread-safe.

### Failure posture

- MinIO unreachable → `runs_total{outcome="error"}` increments, gauges retain their last-known-good value (Prometheus scrape sees "we don't know" as "we know what we knew before" — honest at low frequency, misleading at high frequency, mitigated by a `MetricsAggregatorRunFailureRate` alert on the runs counter).
- Bad JSONL row → skipped silently (per-row unmarshal error swallows; the offending row does not add to any counter). Documented in Not-closed as a candidate for a `pricing_metrics_aggregator_bad_rows_total` counter once the failure surface has real production data behind it.
- Bucket empty → all gauges = 0, run marked `ok`. Legal state.

### Hive-partition-aware scanning

The reader walks `bucket/<prefix>/dt=YYYY-MM-DD/hour=HH/env=<env>/**/*.jsonl.gz`. On a window that straddles an hour boundary (e.g. 09:57:00–10:02:00), both `hour=09/` and `hour=10/` are visited. Objects whose `LastModified` falls outside the window are skipped without opening — one PUT per batch by funnel-sim, so per-object filtering is cheap.

### Deployment shape

Matches the platform's other Go services: distroless static:nonroot image, multi-arch buildx (amd64 + arm64), `Makefile ci-local` runs the standard gates (lint / vet / test / cover / check-adrs), CI job publishes `ghcr.io/helmedeiros/pricing-metrics-aggregator:{main,sha-…,vX.Y.Z}` on main and tag pushes.

## Consequences

### Positive
- Grafana + AlertManager become first-class consumers of the join between markup-decisions and funnel-events — the joined signal was previously only computable by opening a notebook.
- Alerting on business-logic anomalies (conversion collapse, GMV drop) is enabled without operational tooling above what the platform already ships.
- Notebooks and the aggregator are complementary: notebooks answer "why" (hypothesis-driven, high-cardinality), aggregator answers "is this happening now" (fixed cardinality, always-on).

### Negative
- Another compose service to reason about. Mitigated by mirroring the funnel-sim shape — operators reading the compose stack see the same posture (image, healthchecks, /metrics scrape, structured logs).
- The polling latency (up to `--interval`) is a floor on how fresh the metrics can be. Sub-minute alerting is not a use case v0.0.1 addresses.
- v0.0.1 gauge cardinality is intentionally minimal — no arm or segment breakdown. Panel authors expecting arm-level lift in Prometheus today need to wait for ADR-0003.

### Not closed (deferred to follow-on ADRs)

- **ADR-0002 — Window shape.** Tumbling vs sliding, backfill on restart, watermark semantics. v0.0.1 hard-codes tumbling and does not backfill; a restart loses the state of the current window. Follow-on ADR pins the promotion to a durable pattern.
- **ADR-0003 — Metric cardinality caps.** Per-arm + per-segment aggregates would produce a 6-label metric with ~500 unique combinations at typical traffic — ~30k histogram series. Needs a bucketing policy (top-N cardinality carriers + "other" bucket, or drop-below-N thresholds).
- **ADR-0004 — Parquet rollup schema.** Rollups persisted back to a `pricing-metrics-rollup/v1/` MinIO prefix so downstream consumers (analytics notebooks, model-registry challenger flow) read pre-aggregated windows instead of scanning raw events. Trigger: when the analytics notebooks over long time ranges become too slow to iterate on.
- **ADR-0005 — Prometheus exposition rules.** Recording rules that derive rate/histogram views over the last-window gauges + AlertManager rules that fire on run-outcome and business anomalies. Ships alongside a pricing-observability ADR update that references the new alerts.
- **Bad-row counter.** `pricing_metrics_aggregator_bad_rows_total{reason}` per-row unmarshal failures. Trigger: first production data-quality incident.
- **Multi-currency GMV.** v0.0.1 assumes EUR (matches funnel-sim's v1 currency posture). Trigger: when funnel-sim ADR-0003's multi-currency parked item ships.
